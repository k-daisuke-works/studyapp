## 用語間リンク仕様（[[用語名]] 記法）

### 目的
ex フィールド内で他の用語を参照するとき `[[用語名]]` と書くことで、用語間の関連を可視化する。学習者がその場で関連カードに飛べる。

### データ側ルール（Haiku 生成時）
- 他の用語を参照するときは `[[用語名]]` を使う
- `用語名` は `allTerms[].w` と完全一致させる（スペース・記号に注意）
- 例: `→ [[L1キャッシュ]] にない → メモリから取得: 100 ns`

### app.js 実装
1. 初期化時に `TERMS_BY_NAME` Map を作成（`w` → term オブジェクト）
2. `[[用語名]]` を `<span class="term-link" data-w="用語名">用語名</span>` に変換する関数を追加
3. `termHtml()` で `ex` フィールドを表示する際にその関数を通す
4. `.term-link` クリック時のハンドラを追加（対象カードをモーダルまたはスクロールで表示）

### style.css 実装
- `.term-link` を下線リンク風にスタイリング

## 例示（ex）生成フレーム

### 共通ルール

- 前提条件を必ず書く（システム・環境・数値スペックなど）
- 実際の数値・データ・コードブロック・矢印で見せる（文章説明を最小限に）
- 流れが重要な用語は「正常動作 → 問題・異常」の順
- 最後に「だから何が問題か / 何が嬉しいか」を1行で締める
- 他の用語を参照するときは `[[用語名]]` を使う（`allTerms[].w` と完全一致）

### タイプ分類と Few-shot 例

#### タイプ A：現象・誤差系
> 正常値 vs 計算結果の数値比較。結果の差と「だから何が困るか」を示す。

**Few-shot（桁落ち）**
```
【前提条件】工場の品質検査システム。設計値との差が 0.5μm 以内なら合格と判定する。
内部計算は単精度浮動小数点数（有効桁 7 桁）を使っている。

| | 設計値 | 実測値 | 差 | 判定 |
|---|---|---|---|---|
| 本来の値 | 1000000.4 μm | 1000000.8 μm | 0.4 μm | ✅ 合格 |
| 単精度での値 | 1000000 μm | 1000001 μm | 1 μm | ❌ 不合格 |

単精度は 7 桁しか保持できないため、小数点以下が丸められる。
良品なのに廃棄される。
```

---

#### タイプ B：プロトコル・手順系
> 前提条件 → 正常フロー → 異常フロー の順。各ステップに実データ値を入れる。

**Few-shot（DKIM）**
```
【前提条件】amazon.co.jp が DKIM を導入済み。注文確認メールを送信する。

▼ 送信時
① メール本文: "注文番号:12345 金額:3,000円"
        ↓ SHA-256 で計算
② 指紋: "a1b2c3d4"
③ 秘密鍵で暗号化 → 署名: "x9y8z7"
④ ヘッダーに書き込んで送信
   DKIM-Signature: b=x9y8z7; d=amazon.co.jp; ...

▼ 受信時
① ヘッダーから署名 "x9y8z7" を取り出す
② DNS から amazon.co.jp の公開鍵を取得
③ 署名を復号 → "a1b2c3d4"
④ 届いたメール本文を計算 → "a1b2c3d4" → 一致！→ 本物

▼ 攻撃者が改ざんした場合
本文改ざん: "金額:30,000円" → 指紋: "f5e4d3c2"
署名はそのまま "x9y8z7"（秘密鍵がないので更新不可）
復号 → "a1b2c3d4" ≠ "f5e4d3c2" → 改ざん検知 → 迷惑メールへ
```

#### タイプ C：組織・規格系
> 規格なし vs 規格ありの比較。具体的な仕様値を入れる。

**Few-shot（IEEE）**
```
【前提条件】Wi-Fi 規格 IEEE 802.11ac（Wi-Fi 5）が存在する世界としない世界を比較する。

規格がない場合：
  Apple  → 周波数: 5.1GHz  チャンネル幅: 80MHz
  バッファロー → 周波数: 5.3GHz  チャンネル幅: 60MHz
  結果: iPhone ← → バッファロールーター 接続できない

IEEE 802.11ac 規格がある場合：
  全メーカーが同一仕様（5GHz帯・256-QAM・最大6.9Gbps）で実装
  iPhone（Apple）← → バッファロールーター  ✅ 接続できる
  Galaxy（Samsung）← → Cisco ルーター      ✅ 接続できる
```

#### タイプ D：ハードウェア・仕様系
> 前提条件（実機スペック）→ 正常動作の流れ → 数値で効果を示す。

**Few-shot（キャッシュメモリ）**
```
【前提条件】EC サイトが「送料無料ライン: 3,000 円」という設定値を使って、
1 日 10,000 件の注文すべてに「送料がかかるか」をチェックする。

▼ 1 件目
  CPU「送料無料ラインの値が欲しい」→ L1 になし → メモリから取得: 100 ns
  → 次回のために L1 へ自動保存

▼ 2〜10,000 件目
  CPU「送料無料ラインの値が欲しい」→ L1 にある: 1 ns

▼ 合計時間
  キャッシュあり: 100 + (1 × 9,999) =    10,099 ns = 0.01 ms
  キャッシュなし: 100 × 10,000      = 1,000,000 ns = 1 ms

実際のシステムは何百もの値を参照するため、キャッシュの有無が体感できる速度差になる。
```

#### タイプ E：アルゴリズム・処理系
> 入力 → 各ステップの変化 → 出力。「逆算不可」「1文字変わると全然違う」など性質も示す。

**Few-shot（メッセージダイジェスト / SHA-256）**
```
【前提条件】Web サービスがユーザーのパスワードを SHA-256 でハッシュ化して DB に保存する。

▼ 登録時
  入力: "password123" → SHA-256 → "ef92b778bafe771e..."
  ※ 元のパスワードは DB に一切保存しない

▼ ログイン時
  入力: "password123" → SHA-256 → "ef92b778..."
  DB の値と一致 → ✅ ログイン成功

▼ 攻撃者が DB を盗んだ場合
  入手: "ef92b778..." → SHA-256 は逆算不可 → 元のパスワード分からない

▼ 1文字変えると全く別の値
  "password123" → "ef92b778..."
  "Password123" → "0b14d501..."（先頭1文字変えただけで全然違う）
```

#### タイプ F：管理・手法系
> 適用前（混乱・判断できない状態）→ 適用後（整理・判断できる状態）→ 効果を示す。

**Few-shot（リスクマトリックス）**
```
【前提条件】EC サイト（年商 5 億円）のリスク管理会議。
洗い出した 5 件のリスクをどの順番で対策するか決める。

▼ 適用前（一覧だけある状態）
  ・サイバー攻撃 ・決済障害 ・自然災害 ・誤操作 ・配送遅延
  → どれから対策すべきか判断できない

▼ マトリックスに配置
  影響度
  大 │ 自然災害 │ 決済障害 │ サイバー攻撃 ← 最優先
  小 │          │          │ 誤操作・配送遅延
     └──────────────────────────
           低         中        高  → 発生確率

▼ 結果
  🔴 即時: サイバー攻撃  🟠 計画: 決済障害
  🟡 監視: 自然災害      🟢 後回し: 誤操作・配送遅延

5 件を同時対策しようとすると予算が足りない。マトリックスにより「何を先にやるか」が一目で決まる。
```

#### タイプ G：法律・制度系
> 具体的な事実関係 → 法律上の判定 → 結果（違反/適合）。1行の記載の有無で結果が逆転するような例が効果的。

**Few-shot（著作権法）**
```
【前提条件】フリーランスエンジニアが企業 A 社から依頼を受け、
受注管理システムを 200 万円で開発した。契約書に著作権の帰属条項は書かれていない。

▼ 事実関係
  ・フリーランスが自宅で設計・実装（職務著作ではない）
  ・A 社に納品・検収完了
  ・A 社が B 社へシステムをライセンス提供

▼ 著作権法での判定
  著作権の帰属: フリーランス（契約書に記載なし＋職務著作でない）
  A 社がやったこと: 著作権者の許可なく第三者へ配布
  判定: 著作権侵害 ✗

▼ 結果
  フリーランスは A 社に差止請求・損害賠償請求（150 万円相当）が可能

契約書に「著作権は A 社に帰属する」と書いてあれば合法。
一行の記載の有無で結果が逆転する。
```

### Haiku プロンプトテンプレート

```
以下の用語に対して ex / easy / matome フィールドを生成してください。

## 用語情報
w: {用語名}
d: {既存の定義}
example: {既存の使用例}
タイプ: {A〜G}

## ex（例示）の生成ルール
- 前提条件を冒頭に書く（システム・環境・数値など）
- 実際の数値・データ・コードブロック・矢印で見せる
- 流れが重要な用語は「正常 → 異常」の順
- 最後に「だから何が問題か/何が嬉しいか」を1行で締める
- 他の用語を参照するときは [[用語名]] を使う（w フィールドと完全一致）
- タイプ {A〜G} の Few-shot を参照すること

## easy（中学生向け）の生成ルール
- 比喩か身近なものに置き換える（専門用語ゼロ）
- 1〜3文、100字前後
- 「試験では〜」「選択肢を〜」禁止

## matome（まとめ）の生成ルール
- 「〇〇 ＝ △△」の形で一文完結
- 試験直前に見返す想定

## 出力形式（JSON）
{
  "w": "{用語名}",
  "ex": "...",
  "easy": "...",
  "matome": "..."
}
```

### バッチ処理方針

```bash
# 1. 対象テーマを抽出（スクリプト内部では --day N と呼ぶ、UIでは「テーマN」）
python3 scripts/extract_terms.py --type official --day N

# 2. タイプ分類（A〜G）を各用語に付与

# 3. Haiku × 3 並列でパッチ JSON 生成

# 4. patch_terms.py で apply
python3 scripts/patch_terms.py batch.json

# 5. コミット
git add data.json && git commit -m "Add ex/easy/matome for テーマN (N terms)"
```

## 全体チェックコマンド

```python
import json

TEMPLATE_PATTERNS = [
    "選択肢を判別するために覚える", "公式細目で", "試験では",
    "午後問題では", "を一文で説明できるよう", "出題選択肢に現れる公式細目",
]

with open('data.json', encoding='utf-8') as f:
    data = json.load(f)

official = [t for t in data['allTerms'] if t.get('sourceType') == 'official']
remaining = [t for t in official
             if any(p in t.get('d','') + t.get('example','') for p in TEMPLATE_PATTERNS)]
no_easy  = [t for t in official if not t.get('easy')]

print(f"テンプレ残存: {len(remaining)} 件")
print(f"easy 未追加: {len(no_easy)} 件 / {len(official)} 件")
```

## 注意事項

### data.json を直接読まない
5.4MB のため直接 Read するとトークン爆死。必ず `extract_terms.py` 経由で。

### 用語名の罠
- 用語名に余分なスペース・改行が入っている場合がある
- `patch_terms.py` は `sourceDay::w` でキーマッチするため、1文字でも違うとサイレントスキップ
- apply 後に必ず不一致チェックを実行すること

### localStorage スキーマ変更（旧 → 現行）
CLAUDE.md の「localStorage キー一覧」を参照。`ap96_done`（旧）→ `ap96_done_dates`（現行）など大幅変更あり。
