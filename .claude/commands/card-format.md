# カードフォーマット定義

用語カードの各フィールドを書くときはこの定義に従うこと。

---

## d（技術的定義）

試験で問われる正確な定義を1〜2文で書く。

- 規格名・正式名称から始める（例：「IEEE 754 で定めた 32 ビット浮動小数点数。」）
- 構成要素は「A・B・C」形式で列挙する
- ビット数・桁数・範囲など試験に出る数値を必ず含める
- 体言止めで締める

> **必須：結果を導く計算式・関係式がある用語は、その式を必ず本文に書くこと。**
> 「〜を数式で表現する手法」のように式の存在だけ言及して式自体を省略するのは禁止。回帰分析・検定・確率分布・行列演算など「公式を覚えて計算する」ことが試験で問われる用語は、d フィールドの中に実際の式（y = a + bx、log(p/(1-p)) = …、r = Σ…/√… など）を含めること。
> - 例（悪い）：「複数の説明変数と目的変数の関係を数式で表現し、予測する統計手法。」（式が書かれていない）
> - 例（良い）：「目的変数yを複数の説明変数x1〜xnの線形結合 y = β0 + β1x1 + … + βnxn + ε で表し、最小二乗法で係数βiを推定する統計手法。」

**例：**
> IEEE 754 で定めた 32 ビット浮動小数点数。符号 1 ビット・指数 8 ビット・仮数 23 ビットで構成され、有効桁約 7 桁、表現範囲は約 10⁻³⁸〜10³⁸。

---

## easy（やさしく言うと）

情報I履修済みの高校生向けに100〜200字でイメージさせる。

- 「コンピュータが〜するとき、〜だから」と背景・制約から始める
- 「〜という方式のこと」「〜できる」で締める
- 比喩・身近な例で「何が嬉しいか」を伝える
- 「試験では」「選択肢を判別」は禁止

**例：**
> コンピュータが数を保存するとき、メモリの容量が限られているから、小数の桁数を制限しないといけない。単精度は32ビットのメモリを使って、約7桁の精度で数を表す方式のこと。

## example（困り事から始まる理解型説明）

**対象読者：ゲームをやったことがある一般人（IT専門知識なし）。はじめてこの概念に触れる人が読んで腑に落ちる設計にする。**

**文章は一般人でもわかりやすい内容にすること。**
- 専門用語・略語・英単語を文章の中に直接置かない
- 「どうなるか」「なぜ困るか」を日常語で先に説明する
- テーブルの列ヘッダーも日常語にする（「誤差の影響度」→「どれくらいマズいか」）
- 「〇〇を行う」「〇〇が発生する」より「〇〇する」「〇〇になる」で短く書く

> **注：専門用語の対応を括弧で示すこと。**
> 日本語で説明した内容が専門用語に置き換えられる場合は、その直後に `（[[用語名]]）` で専門用語名を添える。「この現象には名前がある」と読者が知れるようにする。
> - 例（良い）：「0と1の並びを左にずらす（[[左シフト]]）だけで2倍になる」
> - 例（悪い）：「[[左シフト]]で2倍になる」（説明なしに用語が先に来る）

**題材は必ずゲーム・ゲーム業界から選ぶこと。**
- 基礎理論・計算系 → ゲームエンジンの内部処理、スコア・グラフィック計算
- ネットワーク・セキュリティ → FF14・原神などのオンラインゲームの通信・認証
- データベース → プレイヤーデータ・アイテム・ランキング管理
- プロジェクト管理 → ゲーム開発プロジェクト（スクエニ・任天堂規模）
- 経営・法務 → ゲーム会社の経営・ガチャ規制・著作権
- ゲームで直接例えにくい場合も、ゲーム業界・ゲーム会社の業務を軸にする

各セクションは**1点につき1行・体言止め**。長くなりそうなら削る。

以下の順番・形式で書く：

### ■ 困り事
なぜこの概念が必要になるのか。1〜2文。専門用語を使わず、中学生が「それ困るな」と感じられる具体的な問題を書く。ゲームの失敗例・限界・不都合から入る。

```
〜をしようとすると〜という問題が起きる。／〜だと〜になってしまう。
```

### ■ これで解決
この概念が何をするか。1文で言い切る。日本語で動作を説明してから、専門用語を `（[[用語名]]）` で添える。

```
〜することで〜できる（[[用語名]]）。
```

### ■ 仕組みを追う
処理の手順をテーブル**1枚**で示す。列ヘッダー・操作の説明は日常語で書き、専門用語は `（[[用語名]]）` で後付け。テーブルの後の `→` で結果の意味を1行で補足する。

テーブルは**各行を `|` で始めること**。行頭に `|` がないとテーブルとして認識されず崩れる。

> **必須：計算式がある用語は「式→係数の出どころ→代入計算」を3ブロックに分けて書くこと。1つの文に詰め込まない。**
> 長い一文に由来説明と計算を混ぜると読みにくい。また、係数の説明を式より先に出すと「何の係数の話か」が読者に伝わらない。以下の**順序・3ブロック構造**で書く：
> 1. **式（1行、最初に出す。係数は記号のまま）**：「◯◯の式：y = a + bx」のように、d フィールドの式を**係数を記号（a, b, w1 など）のまま**ラベル付きで再掲する。**90や−0.5のような具体値をこの時点で式に代入して書かない。** 具体値はこの例のデータで求まった1インスタンスにすぎず、式自体に埋め込むと「これが唯一絶対の式」であるかのように誤読される。
> 2. **係数の出どころ（係数がある場合のみ）**：「今回のデータでは a=90、b=−0.5 と求まる（上の表の3点の実測データ（10日=84%・30日=76%・60日=59%）に最小二乗法をかけて求めた値）。」の形式で、記号に具体値を対応づけながら、**どのデータから求めたかを具体的に名指しする**（「過去の実測データ」のような曖昧な言い方で済ませない）。「当てずっぽうではなく」「仮に置いた数字ではなく」のような否定の前置きは書かない（冗長なだけで情報が増えない）。
>    - **この推定方法名は必ず `[[用語名]]` でリンクすること。** カードが無い状態で名前だけ出すと（`※`注釈が付くだけで）読者は仕組みを確認できない。対象の `data-terms-*.json` に該当カードが無ければ、同じテーマ内に新規 detail カードとして追加する。似た概念の既存カードがある場合はそちらにリンクし、重複カードは作らない（例：「最尤法」は独立カードを作らず既存の `[[最尤推定]]` にリンク）。
>    - **新規カードを追加する前に、用語名がデータセット全体で重複していないか必ず確認すること。** このアプリの用語リンク解決（`TERMS_BY_NAME`）は `w`（用語名）のみをキーにした Map で、**テーマをまたいで同名の用語があると後から読み込まれる方が上書きする**（`DAI_SLUGS` の並び順で決まる。詳細は `app.js` 参照）。同じテーマの `data-terms-*.json` だけでなく `data/data.json` 全体を対象に用語名を検索し、重複が無いことを確認してから新規カードを追加する（実際に「最小二乗法」を新規追加した際、既存のテーマ90カードと名前が衝突し、後から読み込まれる古いカードが表示されてリンクが機能しない事故が起きた）。
>    - **表に載せた数値が、本当にその係数の逆算元になり得るか確認すること。** 変数の数よりデータ点が少ない（例：説明変数3つに対しデータ3件）場合や、係数を求めるのに必要な生データ（例：ロジスティック回帰の0/1結果）がテーブルに存在しない場合は、「表の◯人は元データではなく、求まった式を適用した予測結果の例」であることを明記し、実際にはもっと多くのデータが必要なことも一言添える。都合の悪い制約を隠して「表のデータから求めた」と言い切らない。
> 3. **代入計算（テーブルの行数だけ改行、インデントして列挙）**：具体値を使い `  ラベル → 計算式 = 結果` の形式で、テーブルの各行に対応する代入を1行ずつ書く。1行に複数のケースを詰め込まない。
>
> - 例（悪い）：「係数「90」「−0.5」は当てずっぽうではなく過去の実測データに最小二乗法で求めた値。回帰式：y=90−0.5x。10日→…」（式に具体値を埋め込んでいて一般形が無い／係数の話が式より先／「過去の実測データ」が何を指すか不明／余計な否定フレーズ）
> - 例（良い）：
>   ```
>   回帰式：アクティブ率 = a + b×継続日数
>
>   今回のデータでは a=90、b=−0.5 と求まる（上の表の3点の実測データ（10日=84%・30日=76%・60日=59%）に最小二乗法をかけて求めた値）。
>
>     10日 → 90 − 0.5×10 = 85%
>     30日 → 90 − 0.5×30 = 75%
>     60日 → 90 − 0.5×60 = 60%
>   ```

```
| 手順 | 何をするか | 値 | 意味 |
|---|---|---|---|
| 1 | 〜する（[[用語名]]） | 具体値 | 〜になる |

（式がある場合）◯◯の式：y = 〜

（係数がある場合）この式の係数「〜」は、上の表の〜（具体的なデータを名指し）に〜（[[推定方法]]）をかけて求めた値。

  ケース1 → 〜 = 〜
  ケース2 → 〜 = 〜

→ つまり〜ということ
```

### ■ ゲームでの実例
概念を理解した上で、実際のゲームタイトル・エンジンでどう使われているかを示す。箇条書き2〜3点。

```
- 〜（具体的なゲーム名やエンジン名）では〜のときに使われている
- 〜
```

### ■ 落とし穴
この概念の限界・注意点。箇条書き1〜2点。「〜すると〜になるため〜が必要」の形で。

```
- 〜すると〜になるため〜が必要
```

**クロスリンク：**
- 本文・表のセル内を問わず、登場する用語は `[[用語名]]` でリンクする
- カードあり→クリッカブル、なし→※注釈として自動表示される

**禁止：**
- 専門用語から書き始めない（必ず日本語の説明が先）
- 数値だけ並べて「何が嬉しいか」が分からない説明にしない
- テーブルを2枚以上使わない（1枚に絞る）
- 概念を知っている人向けの「復習リスト」にしない
- 「困り事」を省略してステップから始めない

## ex（図解・例示）

detail カード専用。用語の実体を図や表で可視化する。

- 最初の1文：具体的なシナリオを書く（「スマホの3Dゲームで〜するとき」「銀行で〜を計算するとき」）
- 次の行：`【これが[[用語名]]（補足）】` ヘッダー
- ｜区切りテーブルで構造・対比・ビット列などを表現する（各列にラベルを付ける）
- **テーブルは必ず全角 `｜`（U+FF5C）を使うこと。半角 `|` はテーブルとして認識されず崩れる**
- 空行を挟んで技術的な補足説明を加える
- `[[用語名]]` でクロスリンクを積極的に挿入する

**例：**
```
スマホの3Dゲームでキャラクターの座標を「X=1234.567」として保存するとき、コンピュータは32個の0と1だけで表現する。

【これが[[単精度浮動小数点数]]（32ビット）】
  符号(1ビット) ｜ 指数部(8ビット) ｜ 仮数部(23ビット)
     +          ｜   10001001    ｜  00110100010...
  （プラス）     ｜ （スケール）   ｜ （数字の中身・約7桁）

たった32ビットで約10⁻³⁸〜10³⁸の範囲を表せる。ただし[[仮数]]部が23ビットしかないため有効桁は約7桁。
```
